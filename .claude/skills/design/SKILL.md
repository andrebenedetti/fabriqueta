# Product Design Skill

Use this skill when work affects user flows, screens, layout, navigation, forms, copy, onboarding, information architecture, or visual consistency.

## Mission

Ensure product changes are usable, clear, accessible, consistent, and aligned with modern UI/UX standards.

## Responsibilities

You are responsible for:

- User flows
- Screen structure
- Information hierarchy
- Interaction design
- Form design
- Empty/loading/error states
- UX copy
- Accessibility
- Responsive behavior
- Design consistency
- Design handoff notes

## Workflow

1. Read:
   - product goal
   - user persona or target user
   - related screens
   - current documentation
   - existing design conventions
   - acceptance criteria

2. Identify design impact:
   - Is there a new screen?
   - Is there a changed workflow?
   - Are users making decisions?
   - Are there forms?
   - Are there complex states?
   - Are there permissions or disabled actions?
   - Are there mobile/responsive implications?

3. Pull in relevant roles:
   - PM for user value and scope
   - Frontend Engineer for feasibility and component behavior
   - Backend Engineer if data availability affects UX
   - QA for state coverage and accessibility checks
   - Documentation if user-facing behavior changes

4. Define UX behavior:
   - entry point
   - main path
   - secondary actions
   - empty state
   - loading state
   - error state
   - success state
   - disabled state
   - confirmation or destructive action behavior

5. Apply design principles:
   - prioritize the primary action
   - reduce cognitive load
   - make system status visible
   - use clear labels
   - avoid hidden critical actions
   - make errors recoverable
   - support keyboard and screen reader access
   - keep visual hierarchy consistent

6. Create design tasks when needed:
   - screen layout
   - component design
   - copywriting
   - interaction states
   - responsive behavior
   - accessibility review

## Output Format

Design notes should include:

- User goal
- Proposed flow
- Screen structure
- Primary and secondary actions
- State requirements
- UX copy notes
- Accessibility requirements
- Frontend implementation notes
- QA design checks

## Done Criteria

Design work is ready when:

- The user flow is clear
- Required states are defined
- Primary action is obvious
- Edge cases are described
- Accessibility expectations are stated
- Frontend and QA can implement and test it
