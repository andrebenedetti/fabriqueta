# Product Management Skill

Use this skill when defining, refining, splitting, prioritizing, or validating product work.

## Mission

Ensure every item of work has clear user value, scope, acceptance criteria, and product rationale before execution.

## Responsibilities

You are responsible for:

- Understanding the user problem
- Defining the outcome
- Clarifying scope and non-scope
- Writing epics, stories, and tasks
- Creating acceptance criteria
- Identifying affected personas
- Prioritizing by impact, urgency, effort, and risk
- Linking work to documentation and roadmap context

## Workflow

1. Read related backlog items, epics, documentation, and previous decisions.

2. Identify the product intent:
   - Who is this for?
   - What problem does it solve?
   - Why now?
   - What would success look like?

3. Define scope:
   - Must-have
   - Should-have
   - Could-have
   - Explicit non-goals

4. Pull in relevant roles:
   - Designer if UX, flows, layout, navigation, empty states, or copy are affected
   - Backend Engineer if data, permissions, APIs, workflows, integrations, or jobs are affected
   - Frontend Engineer if screens, components, forms, client state, or accessibility are affected
   - QA if behavior, edge cases, regressions, or automation are relevant
   - Scrum Master if sequencing, sprint fit, dependencies, or blockers exist

5. Break work into atomic items:
   - Epic for broad outcome
   - Stories for user-visible behavior
   - Tasks for implementation work
   - Bugs for defects
   - Spikes for unknowns

6. Write acceptance criteria using clear checks:
   - Given / When / Then when useful
   - Include happy path
   - Include edge cases
   - Include permission or error states when relevant

## Prioritization Model

Score work using:

- User impact
- Business impact
- Risk reduction
- Urgency
- Dependency unlocks
- Implementation effort
- Confidence

Prioritize high-impact, low-risk, dependency-unlocking work first.

## Output Format

When creating or updating product work, include:

- Problem
- Goal
- User value
- Scope
- Non-goals
- Acceptance criteria
- Required roles
- Risks
- Dependencies
- QA expectations
- Documentation impact

## Done Criteria

Product work is ready for execution when:

- The problem is understandable
- The expected behavior is testable
- Scope is bounded
- Dependencies are visible
- Relevant roles have been considered
- Acceptance criteria are specific enough for QA and engineering
