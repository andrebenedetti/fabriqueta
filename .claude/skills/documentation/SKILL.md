# Documentation Skill

Use this skill when product behavior, technical behavior, APIs, workflows, decisions, or sprint outcomes change.

## Mission

Keep documentation accurate, useful, and connected to actual product work.

## Responsibilities

You are responsible for:

- Reading existing docs before creating new ones
- Updating product documentation
- Updating technical documentation
- Writing decision records
- Writing release notes
- Writing sprint and retro recaps
- Avoiding duplicate or stale docs

## Workflow

1. Search existing documentation before writing.

2. Decide documentation type:
   - Product behavior doc
   - User workflow doc
   - API or technical doc
   - Architecture note
   - Decision record
   - QA evidence
   - Sprint recap
   - Retro recap
   - Release note

3. Pull in relevant roles:
   - PM for product intent and user-facing behavior
   - Engineering for technical accuracy
   - QA for verified behavior and known issues
   - Designer for UX flows and screen behavior
   - Scrum Master for sprint/retro context

4. Update existing docs when possible.
   Create new docs only when the topic does not already have a clear home.

5. Every doc should include:
   - purpose
   - current behavior
   - important decisions
   - linked epics/tasks
   - owner role or source of truth
   - last updated context

6. For decisions, record:
   - decision
   - context
   - alternatives considered
   - rationale
   - consequences
   - follow-up tasks

7. For user-facing docs, use plain language.
   For technical docs, include contracts, constraints, and examples.

## Documentation Hygiene

Avoid:

- Duplicating existing docs
- Writing vague summaries
- Leaving outdated behavior uncorrected
- Creating docs with no linked work
- Marking docs complete without source context

## Done Criteria

Documentation is complete when:

- It reflects current behavior
- It links to relevant tasks or epics
- It captures assumptions and decisions
- It is easy for a future agent to use
- It reduces the need to ask a human
